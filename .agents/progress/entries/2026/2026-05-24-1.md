## Summary

Real Doubao M1 job `outputs/3-1` failed on `segment-0007`. Local audio was complete (`13975302 bytes`, `436.725s`), but the public server copy was truncated (`2293760 bytes`, `71.677s`). The old upload path trusted `sftp.put()` without checking the remote file size, so a partial remote WAV could be submitted to Doubao.

The fix adds SFTP post-upload size verification and per-segment retry around the full unit of work: upload, Doubao submit/query, and result conversion.

## Changes

- `SftpUploadService.upload_audio()` now compares local and remote file sizes after upload.
- Remote size mismatch raises retryable `UploadError` and attempts to remove the truncated remote file.
- `transcribe_segments()` retries retryable failures up to `transcription.max_retries + 1` total attempts.
- Segment `retry_count` and `error.detail` now record attempt number, error type, and detail.
- Doubao HTTP timeout and transport errors are wrapped as retryable `DoubaoTranscriptionError`.

## Durable Lesson

For model providers that consume public URLs, upload success must mean the public object is complete, not merely that the upload call returned. The minimum acceptance check is remote size equality immediately after SFTP upload. Retry should wrap the complete segment operation because a failed attempt may leave remote state behind.
