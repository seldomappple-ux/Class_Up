# 网课助手数据协议

## 1. 协议目标

本文定义网课助手从 M1 到 M3 的核心数据结构、文件格式、状态枚举和阶段交接契约。

本协议用于支撑以下开发活动：

- 配置加载
- 任务 manifest 读写
- 音频分段
- 转录服务适配
- 分段转录结果合并
- SRT 与全文文稿生成
- 知识点结构化
- 视频切片
- 学习包目录组织
- 失败恢复和断点重跑

实现代码应以本文为稳定契约。外部模型、ffmpeg 输出或人工修正文件都必须先转换为本文定义的内部结构，再进入后续阶段。

## 2. 通用约定

### 2.1 文件编码

- 文本文件统一使用 UTF-8。
- JSON 文件使用 UTF-8，无 BOM。
- Markdown 文件使用 UTF-8。
- SRT 文件使用 UTF-8。

### 2.2 时间单位

- JSON 内部时间统一使用秒，类型为 number，允许小数。
- 展示用时间码使用 `HH:MM:SS,mmm` 或 `HH-MM-SS`。
- SRT 使用 `HH:MM:SS,mmm`。
- 文件名时间码使用 `HH-MM-SS`。

### 2.3 路径约定

- manifest 中可以保存绝对路径和相对路径。
- `input.video_path` 保存输入视频绝对路径。
- `media.source_video.path` 可以保存输入视频绝对路径。
- `outputs.*` 和中间产物路径优先保存相对输出根目录的路径。
- 所有路径字段必须使用字符串。
- 文件名必须经过非法字符清理。
- 读取或写入相对路径前，必须先基于输出根目录归一化。
- 归一化后的相对路径不得逃逸输出根目录，不允许使用 `../` 指向输出根目录外部。
- 除 `input.video_path` 和 `media.source_video.path` 外，manifest 中的产物路径应保存为相对输出根目录的路径。
- 如果必须记录外部绝对路径，字段说明必须明确标注其不是产物路径。

### 2.4 状态枚举

阶段状态：

- `pending`
- `running`
- `success`
- `failed`
- `skipped`

分段状态：

- `pending`
- `extracting`
- `ready`
- `transcribing`
- `success`
- `failed`
- `skipped`
- `superseded`

其中 `superseded` 表示该分段已被新的子分段替代，不再参与转录、合并或后续阶段读取。

人工复核状态：

- `not_required`
- `required`
- `resolved`

### 2.5 错误对象

所有可恢复错误统一使用 `ErrorInfo`：

```json
{
  "code": "TRANSCRIPTION_TIMEOUT",
  "message": "Transcription request timed out.",
  "detail": "Request exceeded timeout_seconds=120.",
  "retryable": true,
  "occurred_at": "2026-05-18T10:30:00+08:00"
}
```

字段要求：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `code` | string | 是 | 稳定错误码 |
| `message` | string | 是 | 简短错误说明 |
| `detail` | string | 否 | 排查细节，不包含密钥 |
| `retryable` | boolean | 是 | 是否允许自动重试 |
| `occurred_at` | string | 是 | ISO 8601 时间 |

错误码规则：

- `code` 必须使用大写英文、数字和下划线。
- 错误码应稳定，不得把动态参数、文件名、HTTP 状态说明拼入 `code`。
- 阶段相关错误码建议使用阶段或模块前缀，例如 `CONFIG_`、`FFMPEG_`、`SEGMENT_`、`TRANSCRIPTION_`、`ANALYSIS_`、`CLIP_`。
- 具体错误上下文写入 `message` 或 `detail`，不得写入密钥、Cookie、Token 或带签名的临时 URL。

最小错误码集合：

| 错误码 | 场景 | 默认可重试 |
|---|---|---|
| `CONFIG_INVALID` | 配置缺失、类型错误或取值越界 | 否 |
| `FFMPEG_NOT_FOUND` | 未找到 ffmpeg 或 ffprobe | 否 |
| `FFMPEG_FAILED` | ffmpeg 或 ffprobe 执行失败 | 否 |
| `SEGMENT_TOO_LARGE` | 音频分段超过上传限制且重切后仍失败 | 否 |
| `TRANSCRIPTION_TIMEOUT` | 转录服务请求超时 | 是 |
| `TRANSCRIPTION_FAILED` | 转录服务返回失败或不可解析结果 | 是 |
| `ANALYSIS_SCHEMA_INVALID` | 文本分析输出不符合知识点结构 | 否 |
| `CLIP_EXPORT_FAILED` | 视频切片导出失败 | 否 |

## 3. 配置协议

配置文件建议为 `config/config.example.yaml`。

```yaml
project:
  output_root: outputs

media:
  audio_sample_rate: 16000
  audio_channels: 1
  audio_format: wav
  segment_seconds: 600
  overlap_seconds: 30

transcription:
  provider: mock
  endpoint: ""
  model: ""
  auth_type: api_key
  api_key_env: CLASS_UP_TRANSCRIPTION_API_KEY
  resource_id: ""
  upload_limit_mb: 25
  concurrency: 3
  timeout_seconds: 120
  max_retries: 3
  poll_interval_seconds: 1
  max_poll_seconds: 600
  request:
    enable_itn: true
    enable_punc: true
    enable_ddc: false
    enable_speaker_info: false
    enable_channel_split: false
    show_utterances: true
    vad_segment: true
    sensitive_words_filter: ""

upload:
  provider: none
  host_env: CLASS_UP_UPLOAD_HOST
  port: 22
  username_env: CLASS_UP_UPLOAD_USER
  private_key_path_env: CLASS_UP_UPLOAD_KEY_PATH
  remote_dir: /var/www/class-up/audio/
  public_url_base: https://boneorbit.com/class-up/audio/

analysis:
  provider: mock
  endpoint: ""
  model: ""
  auth_type: api_key
  api_key_env: CLASS_UP_ANALYSIS_API_KEY
  timeout_seconds: 180
  max_retries: 2

output:
  keep_intermediate: true
  overwrite: false
```

配置要求：

- `provider` 可以是 `mock`、`doubao`、`local`、`remote` 或后续新增的适配器名称。
- `provider` 为 `doubao` 时，具体协议以 `docs/M1开发阶段/豆包录音识别接入协议.md` 为准。
- 真实密钥只从环境变量读取。
- 配置文件不得保存真实 API Key。
- `model` 可以为空；为空时由适配器自行选择默认模型，但业务流程不得硬编码模型名。
- `segment_seconds` 必须大于 `overlap_seconds`。
- `concurrency` 必须大于等于 1。
- `upload.provider` 为 `sftp` 时，上传配置只保存环境变量名和非敏感路径，不保存 SSH 私钥内容。
- Web UI 可以把用户在页面填写的豆包 API Key 写入当前后端进程的 `CLASS_UP_DOUBAO_API_KEY` 环境变量，但不得写入配置文件、manifest、原始响应或日志。
- Web UI 选择 `doubao` 时，后端应默认使用 `endpoint=https://openspeech.bytedance.com`、`model=bigmodel`、`resource_id=volc.seedasr.auc`，并将 `upload.provider` 设为 `sftp`。
- Web UI 不采集 SFTP 配置；`CLASS_UP_UPLOAD_HOST`、`CLASS_UP_UPLOAD_USER`、`CLASS_UP_UPLOAD_KEY_PATH` 必须来自 `.env` 或系统环境变量。

## 4. 输出目录协议

标准输出目录：

```text
课程名/
  manifest.json
  full_transcript.txt
  full_subtitle.srt
  knowledge_base.md
  knowledge_points.json
  intermediate/
    audio/
      full_audio.wav
    segments/
      segment-0001.wav
      segment-0002.wav
    transcription/
      segment-0001.json
      segment-0002.json
    review/
      overlap-review.json
      analysis-error.json
  clips/
    01_梯度下降原理_00-12-30/
      clip.mp4
      subtitle.srt
      notes.md
```

目录说明：

| 路径 | 阶段 | 说明 |
|---|---|---|
| `manifest.json` | 全阶段 | 任务主记录 |
| `full_transcript.txt` | M1 | 完整文稿 |
| `full_subtitle.srt` | M1 | 完整字幕 |
| `knowledge_points.json` | M2 | 知识点结构化数据 |
| `knowledge_base.md` | M2 | 知识库版文稿 |
| `intermediate/audio/` | M1 | 标准化完整音频 |
| `intermediate/segments/` | M1 | 音频分段 |
| `intermediate/transcription/` | M1 | 每段转录 JSON |
| `intermediate/review/` | M1-M3 | 人工复核或错误修正文件 |
| `clips/` | M3 | 知识点切片学习包 |

### 4.1 M1 音频准备输出

`class-up audio` 归属于 M1 音频准备能力，用于将单个视频文件转换为标准化音频文件。

M1 音频准备有两种执行形态：

- 完整 M1 流水线：标准化音频写入标准输出目录的 `intermediate/audio/`，并记录到 `manifest.media.normalized_audio`。
- 单独音频转换命令：只执行 M1 音频准备中的转换动作，输出到用户指定路径，不推进分段、转录、合并阶段。

命令约定：

- 默认输出格式为 `wav`。
- 默认采样率为 `16000 Hz`。
- 默认声道数为 `1`。
- `--output` 表示完整输出音频文件路径。
- `--output-dir` 表示输出目录，文件名默认使用输入视频同名、扩展名为 `.wav`。
- 未指定 `--overwrite` 时，不覆盖已存在的输出文件。
- 单独音频转换命令不写入 `segments[]`，不生成转录结果。

## 5. Manifest 协议

`manifest.json` 是任务的主状态文件。

### 5.1 顶层结构

```json
{
  "schema_version": "1.0",
  "task_id": "20260518-103000-class-up",
  "created_at": "2026-05-18T10:30:00+08:00",
  "updated_at": "2026-05-18T10:45:00+08:00",
  "status": "running",
  "input": {},
  "config_summary": {},
  "stages": {},
  "media": {},
  "segments": [],
  "outputs": {},
  "clips": [],
  "reviews": [],
  "errors": []
}
```

顶层字段要求：

| 字段 | 类型 | 必填 | 可空 | 说明 |
|---|---|---|---|---|
| `schema_version` | string | 是 | 否 | manifest 协议版本 |
| `task_id` | string | 是 | 否 | 稳定任务 ID |
| `created_at` | string | 是 | 否 | ISO 8601 创建时间 |
| `updated_at` | string | 是 | 否 | 最近一次 manifest 写入时间 |
| `status` | string | 是 | 否 | 任务顶层状态，取阶段状态枚举 |
| `input` | object | 是 | 否 | 输入信息 |
| `config_summary` | object | 是 | 否 | 非敏感配置摘要 |
| `stages` | object | 是 | 否 | M1-M3 阶段状态 |
| `media` | object | 是 | 否 | 媒体信息，未知字段可先写 `null` |
| `segments` | array | 是 | 否 | M1 分段列表，未生成前为空数组 |
| `outputs` | object | 是 | 否 | 全局输出路径，未生成前对应字段为 `null` |
| `clips` | array | 是 | 否 | M3 切片产物记录，未生成前为空数组 |
| `reviews` | array | 是 | 否 | 人工复核记录，未生成前为空数组 |
| `errors` | array | 是 | 否 | 任务级错误记录，未发生错误前为空数组 |

状态同步规则：

- `status` 表示整个任务的当前状态，不替代 `stages.*.status`。
- 任一阶段为 `running` 时，顶层 `status` 必须为 `running`。
- 任一必须阶段为 `failed` 且流程不能继续时，顶层 `status` 必须为 `failed`。
- M1-M3 均为 `success` 时，顶层 `status` 必须为 `success`。
- 尚未开始或只初始化 manifest 时，顶层 `status` 可以为 `pending`。
- 跳过某阶段时，必须在对应阶段 `status` 写入 `skipped`，并在阶段 `error` 或 `reviews[]` 说明原因。

时间更新规则：

- 创建 manifest 时写入 `created_at` 和 `updated_at`。
- 每次写入 manifest 时必须更新 `updated_at`。
- 阶段从 `pending` 进入 `running` 时写入 `started_at`。
- 阶段进入 `success`、`failed` 或 `skipped` 时写入 `finished_at`。
- 阶段尚未开始时，`started_at`、`finished_at` 和 `error` 均为 `null`。

### 5.7 Clips

M3 生成的切片产物记录保存在 `manifest.clips[]`。

```json
{
  "knowledge_id": "kp-0001",
  "index": 1,
  "title": "梯度下降原理",
  "start": 750.0,
  "end": 1125.0,
  "clip_path": "clips/01_梯度下降原理_00-12-30/clip.mp4",
  "subtitle_path": "clips/01_梯度下降原理_00-12-30/subtitle.srt",
  "notes_path": "clips/01_梯度下降原理_00-12-30/notes.md",
  "error": null
}
```

字段要求：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `knowledge_id` | string | 是 | 对应 `knowledge_points.json.items[].knowledge_id` |
| `index` | integer | 是 | 对应知识点序号 |
| `title` | string | 是 | 对应知识点标题 |
| `start` | number | 是 | 原视频绝对起点秒 |
| `end` | number | 是 | 原视频绝对终点秒 |
| `clip_path` | string/null | 是 | 切片视频路径 |
| `subtitle_path` | string/null | 是 | 切片字幕路径 |
| `notes_path` | string/null | 是 | 切片笔记路径 |
| `error` | object/null | 是 | 切片失败错误对象 |

`knowledge_points.json` 仍然是 M3 切片时间范围真源；`manifest.clips[]` 只记录 M3 产物路径和执行结果。

## 6. 音频分段协议

分段记录保存在 `manifest.segments[]`。

```json
{
  "segment_id": "segment-0001",
  "parent_segment_id": null,
  "index": 1,
  "status": "success",
  "start": 0.0,
  "end": 600.0,
  "overlap_previous_seconds": 0,
  "overlap_next_seconds": 30,
  "audio_path": "intermediate/segments/segment-0001.wav",
  "size_bytes": 19200000,
  "upload_limit_mb": 25,
  "transcription_path": "intermediate/transcription/segment-0001.json",
  "retry_count": 0,
  "error": null
}
```

字段要求：

| 字段 | 类型 | 必填 | 时间基准 | 说明 |
|---|---|---|---|---|
| `segment_id` | string | 是 | 不适用 | 稳定分段 ID |
| `parent_segment_id` | string/null | 是 | 不适用 | 超限重切时指向原分段 |
| `index` | integer | 是 | 不适用 | 合并排序主键 |
| `status` | string | 是 | 不适用 | 分段状态 |
| `start` | number | 是 | 原视频绝对时间 | 原视频绝对起点秒 |
| `end` | number | 是 | 原视频绝对时间 | 原视频绝对终点秒 |
| `overlap_previous_seconds` | number | 是 | 时长 | 与前段重叠秒数 |
| `overlap_next_seconds` | number | 是 | 时长 | 与后段重叠秒数 |
| `audio_path` | string | 是 | 不适用 | 分段音频路径 |
| `size_bytes` | integer | 是 | 不适用 | 文件大小 |
| `upload_limit_mb` | number | 是 | 不适用 | 当前分段使用的上传限制 |
| `transcription_path` | string/null | 是 | 不适用 | 转录结果路径 |
| `retry_count` | integer | 是 | 不适用 | 当前分段重试次数 |
| `error` | object/null | 是 | 不适用 | 错误对象 |

时间基准说明：

- `manifest.segments[].start` 和 `manifest.segments[].end` 永远使用原视频绝对时间。
- 分段转录结果中的 `items[].start` 和 `items[].end` 永远使用分段内相对时间。
- 合并阶段必须用分段绝对起点加上转录 item 相对时间，生成原视频绝对时间。

排序规则：

- 合并阶段必须按 `index` 升序排序，不得依赖 `segment_id` 字符串排序。
- `index` 是合并排序的唯一协议字段。
- `segment_id` 只作为稳定标识和文件命名依据。
- 子分段必须拥有独立 `index`，用于表达其在全局时间轴中的顺序。
- 当父分段被重切为多个子分段时，父分段状态改为 `superseded`，子分段按原父分段位置插入新的连续 `index`。
- manifest 写入后应保证所有可参与合并的分段 `index` 连续递增。

超限重切规则：

- 原分段不得继续作为可转录分段参与后续阶段。
- 原分段状态必须改为 `superseded`。
- 新子分段使用稳定 ID，例如 `segment-0003a`、`segment-0003b`。
- 子分段必须记录 `parent_segment_id`。
- manifest 中必须能区分旧分段已废弃和新分段可用。
- 后续合并只读取状态为 `success` 且有 `transcription_path` 的分段。

## 7. 分段转录结果协议

每个分段转录结果保存为 `intermediate/transcription/{segment_id}.json`。

```json
{
  "schema_version": "1.0",
  "segment_id": "segment-0001",
  "source_audio": "intermediate/segments/segment-0001.wav",
  "time_base": "segment_relative",
  "language": "zh",
  "provider": "mock",
  "model": "",
  "items": [
    {
      "item_id": "segment-0001-item-0001",
      "start": 0.0,
      "end": 3.2,
      "text": "本节课我们介绍梯度下降。",
      "confidence": null
    }
  ],
  "raw_output_path": null,
  "error": null
}
```

关键约束：

- `time_base` 固定为 `segment_relative`。
- `items[].start` 和 `items[].end` 是分段内相对时间，不是原视频绝对时间。
- 合并阶段负责加上分段在原视频中的起点。
- `end` 必须大于等于 `start`。
- `text` 不得为空字符串；空白内容应在适配器层过滤。
- 外部模型原始返回可以保存为 `raw_output_path`，但不得影响后续阶段读取内部结构。
- 每段转录 JSON 必须使用顶层 `items[]` 表达转录片段。
- 禁止把旧版顶层 `start`、`end`、`text` 结构作为内部真源；适配器必须先转换为本节结构再写入后续流程。

## 8. 合并转录结果协议

M1 合并后形成两个最终文件：

- `full_subtitle.srt`
- `full_transcript.txt`

同时可选生成内部合并 JSON：

```json
{
  "schema_version": "1.0",
  "time_base": "video_absolute",
  "items": [
    {
      "item_id": "merged-000001",
      "source_segment_id": "segment-0001",
      "start": 0.0,
      "end": 3.2,
      "text": "本节课我们介绍梯度下降。",
      "overlap_policy": "none"
    }
  ],
  "reviews": []
}
```

重叠区处理规则：

- 默认保留后段重叠区，前段仅丢弃与后段时间重叠的尾部片段。
- 禁止同时裁掉前段尾部和后段头部导致重叠区内容缺失。
- 无法安全判断时优先保留内容。
- 需要人工复核时写入统一 `reviews[]` 对象，并在 manifest 中记录。
- 不得另行引入散落布尔字段表达复核需求。

Review 示例：

```json
{
  "review_id": "overlap-0001",
  "type": "overlap",
  "status": "required",
  "segment_ids": ["segment-0001", "segment-0002"],
  "start": 570.0,
  "end": 600.0,
  "message": "Overlap region could not be safely deduplicated.",
  "resolution": null,
  "resolved_at": null
}
```

Review 字段要求：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `review_id` | string | 是 | 稳定复核 ID |
| `type` | string | 是 | 复核类型，例如 `overlap`、`analysis_schema`、`manual_fix` |
| `status` | string | 是 | 人工复核状态 |
| `segment_ids` | array | 是 | 相关分段 ID；非分段类复核可为空数组 |
| `start` | number/null | 是 | 原视频绝对起点秒；不适用时为 `null` |
| `end` | number/null | 是 | 原视频绝对终点秒；不适用时为 `null` |
| `message` | string | 是 | 需要复核的原因 |
| `resolution` | string/null | 是 | 人工处理结论，未处理时为 `null` |
| `resolved_at` | string/null | 是 | ISO 8601 处理时间，未处理时为 `null` |

`reviews[]` 可以同时出现在内部合并 JSON 和 manifest 中；manifest 中的记录用于阶段交接和断点恢复。

## 9. SRT 协议

SRT 输出示例：

```srt
1
00:00:00,000 --> 00:00:03,200
本节课我们介绍梯度下降。

2
00:00:03,200 --> 00:00:06,500
它是一种常见的优化方法。
```

要求：

- 编号从 1 开始连续递增。
- 时间码必须单调递增。
- 单条字幕 `end` 必须大于 `start`。
- 不允许出现负时间。
- 字幕文本使用合并后的 `text`。

## 10. 全文文稿协议

`full_transcript.txt` 为纯文本文件。

要求：

- 按合并后的时间顺序输出。
- 默认每条合并项一行。
- 不写入时间码。
- 不写入模型服务内部调试信息。

示例：

```text
本节课我们介绍梯度下降。
它是一种常见的优化方法。
```

## 11. 知识点协议

M2 输出 `knowledge_points.json`。

```json
{
  "schema_version": "1.0",
  "source": {
    "transcript": "full_transcript.txt",
    "subtitle": "full_subtitle.srt"
  },
  "time_base": "video_absolute",
  "items": [
    {
      "index": 1,
      "knowledge_id": "kp-0001",
      "title": "梯度下降原理",
      "start": 750.0,
      "end": 1125.0,
      "keywords": ["梯度", "学习率", "损失函数"],
      "summary": "本段介绍梯度下降如何通过迭代更新参数降低损失。",
      "content": "本知识点正文...",
      "confidence": null,
      "review_status": "not_required"
    }
  ],
  "error": null
}
```

约束：

- `time_base` 固定为 `video_absolute`。
- `items[].index` 从 1 开始递增。
- `start` 必须小于 `end`。
- `start` 和 `end` 不能超出视频总时长。
- `title` 必须经过文件名安全清理后才能用于目录名。
- `keywords` 可以为空数组，但不能为 null。
- 模型输出不符合结构时，不直接进入 M3。

## 12. 知识库 Markdown 协议

M2 输出 `knowledge_base.md`。

```markdown
## 01. 梯度下降原理

时间：00:12:30 - 00:18:45
关键词：梯度、学习率、损失函数

摘要：
本段介绍梯度下降如何通过迭代更新参数降低损失。

正文：
本知识点正文...
```

要求：

- 每个知识点一个二级标题。
- 标题序号与 `knowledge_points.json.items[].index` 一致。
- 时间范围来自 `knowledge_points.json`。
- Markdown 仅作为阅读和知识库导入文件，不作为 M3 的机器读取真源。
- M3 只读取 `knowledge_points.json`。

## 13. 视频切片协议

M3 根据 `knowledge_points.json` 生成切片目录。

切片目录命名：

```text
{index}_{safe_title}_{HH-MM-SS}/
```

示例：

```text
01_梯度下降原理_00-12-30/
  clip.mp4
  subtitle.srt
  notes.md
```

`notes.md` 内容来自对应知识点：

```markdown
# 梯度下降原理

时间：00:12:30 - 00:18:45
关键词：梯度、学习率、损失函数

## 摘要

本段介绍梯度下降如何通过迭代更新参数降低损失。

## 正文

本知识点正文...
```

切片要求：

- `clip.mp4` 时间范围来自知识点 `start` / `end`。
- `subtitle.srt` 只包含该知识点时间范围内的字幕。
- 切片内 `subtitle.srt` 时间码必须重置为切片相对时间，从 `00:00:00,000` 开始。
- 原视频绝对时间只保留在 `notes.md` 和 manifest 的切片记录中。
- `notes.md` 只包含该知识点内容。
- 切片目录名必须经过非法字符清理。

## 14. 文件名安全协议

文件名清理规则：

- 删除或替换 Windows 非法字符：`< > : " / \ | ? *`
- 去除首尾空白。
- 将连续空白折叠为单个空格或下划线。
- 空标题使用 `untitled`。
- 文件名长度超过限制时截断，但保留序号和时间码。

推荐目录名格式：

```text
01_梯度下降原理_00-12-30
```

## 15. 阶段交接契约

### 15.1 M1 -> M2

M2 启动前必须存在：

- `manifest.json`
- `full_subtitle.srt`
- `full_transcript.txt`
- manifest 中 `stages.m1.status == "success"`

M2 可读取：

- `full_transcript.txt`
- `full_subtitle.srt`
- manifest 中的视频总时长和配置摘要

### 15.2 M2 -> M3

M3 启动前必须存在：

- `knowledge_points.json`
- `knowledge_base.md`
- manifest 中 `stages.m2.status == "success"`

M3 只以 `knowledge_points.json` 作为切片时间范围真源。

### 15.3 M3 完成

M3 完成后必须更新：

- manifest 中 `stages.m3.status`
- manifest 中 `outputs.clips_root`
- manifest 中 `clips[]` 的每个知识点切片产物路径

## 16. 版本管理

所有 JSON 顶层结构必须包含：

```json
{
  "schema_version": "1.0"
}
```

兼容规则：

- 小版本新增字段不得破坏旧字段含义。
- 删除字段或改变字段语义必须提升主版本。
- 读取端应忽略未知字段。
- 写入端必须写入当前协议要求的必填字段。

## 17. 最小校验清单

实现完成后，至少校验：

- 配置文件不包含真实密钥。
- manifest 不包含真实密钥。
- 所有 JSON 文件是合法 UTF-8 JSON。
- 分段 `start < end`。
- 转录 item `start <= end`。
- 合并后字幕时间码单调递增。
- 知识点时间范围不超出视频总时长。
- 切片目录名不包含 Windows 非法字符。
- M2 不读取 Markdown 作为机器真源。
- M3 不绕过 `knowledge_points.json` 直接解析自然语言文稿。
