# M1 开发计划：工程基础与转录流水线

## 1. 阶段目标

M1 的目标是完成从本地视频文件到完整字幕和全文文稿的端到端流水线。

输入：

- 本地视频文件
- 输出目录
- 模型服务配置文件

输出：

- `manifest.json`
- `full_subtitle.srt`
- `full_transcript.txt`
- 音频中间产物目录
- 分段转录中间产物

M1 完成后，项目应能够在 Windows 本机用命令行处理一个 30 分钟以内的视频，并在失败后支持从已完成分段继续重跑。

## 2. 本阶段不做

M1 不处理以下内容：

- 知识点分段
- 知识库 Markdown
- 视频切片
- 学习包最终目录整理
- Gradio UI
- 固定某个具体模型名称
- 在线课程平台下载或账号登录

## 4. 配置要求

M1 配置文件至少覆盖：

```yaml
project:
  output_root: outputs

media:
  audio_sample_rate: 16000
  audio_channels: 1
  audio_format: wav
  segment_seconds: 600
  overlap_seconds: 30

transcription:
  provider: local_or_remote_provider
  endpoint: ""
  model: ""
  auth_type: api_key
  api_key_env: CLASS_UP_TRANSCRIPTION_API_KEY
  upload_limit_mb: 25
  concurrency: 3
  timeout_seconds: 120
  max_retries: 3
```

要求：

- 不把真实 API Key 写入配置文件。
- 配置文件只记录环境变量名。
- 具体模型名称允许配置，但业务代码不能硬编码。
- 音频格式由配置决定，默认使用 `wav`。
- `config.example.yaml` 必须可以直接作为无密钥 mock 流程的配置模板使用。

## 5. Manifest 要求

M1 的数据结构、状态枚举、路径规则和阶段交接以 `docs/数据协议.md` 为准；本计划只描述 M1 实施任务，不另行定义冲突结构。

`manifest.json` 至少记录：

- 输入视频绝对路径
- 输出目录绝对路径
- 配置摘要，不包含密钥原文
- 阶段状态
- 音频文件路径
- 分段列表
- 每个分段的状态、音频路径、转录结果路径、错误信息
- 最终字幕和文稿路径

阶段状态取值：

- `pending`
- `running`
- `success`
- `failed`
- `skipped`

断点恢复要求：

- 已完成分段不重复调用转录服务。
- 失败分段允许单独重试。
- 合并阶段只能读取 `status == "success"` 且有 `transcription_path` 的分段结果。
- 如果分段因文件大小超限被重切，父分段必须标记为 `superseded`，新的子分段写入独立 `index` 和 `parent_segment_id`，避免断点恢复时同时读取旧分段和新分段。
- 所有可参与合并的分段必须保持 `index` 连续递增。

## 6. 执行步骤

### Step 1：初始化工程骨架

任务：

- [ ] 创建 `src/class_up/` 包结构
- [ ] 创建基础 `__init__.py`
- [ ] 创建 CLI 入口文件
- [ ] 创建配置、manifest、media、transcription、utils 模块

验收：

- Python 能正常导入 `class_up`
- CLI 能输出帮助信息

### Step 2：配置加载与路径初始化

任务：

- [ ] 读取 YAML 配置
- [ ] 读取 `.env` 中的密钥环境变量
- [ ] 校验必填配置
- [ ] 创建标准输出目录
- [ ] 初始化 `manifest.json`

验收：

- 缺少必填配置时给出明确错误
- 不在日志和 manifest 中输出密钥原文
- 输出目录可以重复初始化且不会覆盖原视频

### Step 3：ffmpeg 封装

任务：

- [ ] 检查 ffmpeg 是否可执行
- [ ] 封装命令执行函数
- [ ] 捕获 stdout、stderr、退出码
- [ ] 将 ffmpeg 错误写入 manifest

验收：

- ffmpeg 不存在时能明确报错
- ffmpeg 执行失败时保留错误信息

### Step 4：音频提取与分段

任务：

- [ ] 用 ffprobe 获取视频时长，写入 manifest
- [ ] 从视频提取音频
- [ ] 标准化采样率和声道
- [ ] 按配置时长切段，保留配置化重叠区
- [ ] 切段后检查每段文件大小；若超过上传限制，自动将该段时长减半后重切，最多重试 2 次，仍超限则写入 manifest 并报错
- [ ] 超限重切后使用稳定子分段 ID，例如 `segment-0003a`、`segment-0003b`，并在 manifest 中记录 `parent_segment_id`
- [ ] 将分段列表（含起止时间、文件路径、文件大小）写入 manifest

验收：

- 分段文件存在且可被 ffmpeg 识别
- 分段起止时间连续，重叠区符合配置
- 所有分段文件大小不超过配置的上传限制
- manifest 记录视频总时长
- 超限重切不会留下可被后续阶段误读的旧分段记录

### Step 5：转录适配接口与 mock 服务

任务：

- [ ] 定义转录服务接口
- [ ] 定义统一分段转录结果结构，使用顶层 `items[]` 表达转录片段
- [ ] 实现 mock 转录服务
- [ ] 保存每段转录结果 JSON

验收：

- mock 服务可跑通所有分段
- 每段结果包含 `schema_version`、`segment_id`、`source_audio`、`time_base`、`items[]`、`error`
- `time_base` 固定为 `segment_relative`
- `items[].start` / `items[].end` 均为分段内相对时间，不直接写成原视频绝对时间
- 不把旧版顶层 `start`、`end`、`text` 结构作为内部真源

### Step 6：首个真实转录服务适配器

任务：

- [ ] 根据配置选择转录服务适配器
- [ ] 读取鉴权信息
- [ ] 实现超时、重试和并发控制
- [ ] 记录每段转录状态

验收：

- 真实服务失败不会导致任务无记录退出
- 重跑时跳过已成功分段
- 并发数由配置控制

### Step 7：合并与字幕生成

任务：

- [ ] 按 manifest 中可参与合并分段的 `index` 升序读取转录结果后再合并，不按 `segment_id` 排序
- [ ] 修正每段时间戳偏移（加上该段在原始音频中的起始时间）
- [ ] 重叠区处理策略：默认保留后段重叠区，前段仅丢弃与后段时间重叠的尾部片段；禁止同时裁掉前段尾部和后段头部导致重叠区内容丢失
- [ ] 对无法安全判断的重叠区，优先保留内容并在 manifest 的统一 `reviews[]` 对象中记录复核需求
- [ ] 生成 `full_subtitle.srt`
- [ ] 生成 `full_transcript.txt`

验收：

- SRT 编号连续
- SRT 时间码单调递增，无负值
- 文稿按转录顺序输出
- 重叠区处理不得造成整段重叠内容缺失
- 无法安全去重的重叠区会写入 `reviews[]`，不使用散落布尔字段
- 合并结果路径写入 manifest

### 7.3 手工验收

准备一个 30 分钟以内样例视频，执行完整 M1 流程。

验收检查：

- 输出 `full_subtitle.srt`
- 输出 `full_transcript.txt`
- manifest 最终状态为 `success`
- 字幕时间码无乱序
- 日志和 manifest 不包含 API Key 原文

## 8. 完成标准

M1 只有同时满足以下条件，才视为完成：

- CLI 可用
- `pyproject.toml`、`.gitignore` 和测试命令可用
- 配置加载可用
- ffmpeg 检测、音频提取、音频切段可用
- mock 转录服务端到端通过
- 首个真实转录服务适配器完成
- SRT 和全文文稿可生成
- manifest 支持失败记录和断点恢复
- 单元测试和集成测试通过
- 30 分钟以内样例视频手工验收通过

## 9. 风险与处理

| 风险 | 处理 |
|---|---|
| 转录服务时间戳格式不一致 | 适配器层转换为统一内部结构 |
| 单段音频超过上传限制 | 分段后检查文件大小，必要时缩短分段时长，并用稳定子分段 ID 更新 manifest |
| ffmpeg 命令失败 | 集中在 `media/ffmpeg.py` 捕获并写入 manifest |
| 重叠区重复文本难以完全去除 | 先实现保守去重，宁可保留重复，也不能丢失重叠区内容 |
| API 限流 | 并发数、重试次数、超时全部配置化 |
| 断点恢复误跳过失败段 | manifest 明确区分 `success`、`failed`、`skipped` |

## 10. 执行顺序

推荐按以下顺序推进：

1. Step 1 工程骨架
2. Step 2 配置加载与路径初始化
3. Step 3 ffmpeg 封装
4. Step 4 音频提取与分段
5. Step 5 mock 转录服务
6. Step 7 合并与字幕生成（先用 mock 服务闭环，验证合并逻辑）
7. Step 6 首个真实转录服务适配器（替换 mock，接入本地或远程模型服务）
8. 测试补齐与手工验收

先用 mock 服务闭环，把工程问题和模型服务问题分开排查，闭环后再替换为真实转录服务适配器。
