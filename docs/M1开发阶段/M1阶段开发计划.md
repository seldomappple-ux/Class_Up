# M1 开发计划：工程基础与转录流水线

## 1. 阶段目标

M1 的目标是完成从本地视频文件到完整字幕和全文文稿的端到端流水线。

输入：

- 本地视频文件
- 输出目录
- 模型服务配置文件

输出：

- `manifest.json`
- `full_subtitle.srt`
- `full_transcript.txt`
- 音频中间产物目录
- 分段转录中间产物

M1 完成后，项目应能够在 Windows 本机用命令行处理一个 30 分钟以内的视频，并在失败后支持从已完成分段继续重跑。

一键视频转音频归入 M1 的音频准备子功能。它用于单独执行和验收 M1 的视频转音频能力，可用于快速检查 ffmpeg 环境、准备模型调试音频或人工验收音频质量；完整 M1 流水线仍负责初始化 manifest、音频分段、转录、合并和最终输出。

## 2. 本阶段不做

M1 不处理以下内容：

- 知识点分段
- 知识库 Markdown
- 视频切片
- 学习包最终目录整理
- Gradio UI
- 固定某个具体模型名称
- 在线课程平台下载或账号登录

## 4. 配置要求

M1 配置文件至少覆盖：

```yaml
project:
  output_root: outputs

media:
  audio_sample_rate: 16000
  audio_channels: 1
  audio_format: wav
  segment_seconds: 600
  overlap_seconds: 30

transcription:
  provider: mock
  endpoint: ""
  model: ""
  auth_type: api_key
  api_key_env: CLASS_UP_TRANSCRIPTION_API_KEY
  resource_id: ""
  upload_limit_mb: 25
  concurrency: 3
  timeout_seconds: 120
  max_retries: 3
  poll_interval_seconds: 1
  max_poll_seconds: 600

upload:
  provider: none
  host_env: CLASS_UP_UPLOAD_HOST
  port: 22
  username_env: CLASS_UP_UPLOAD_USER
  private_key_path_env: CLASS_UP_UPLOAD_KEY_PATH
  remote_dir: /var/www/class-up/audio/
  public_url_base: https://boneorbit.com/class-up/audio/
```

要求：

- 不把真实 API Key 写入配置文件。
- 配置文件只记录环境变量名。
- 具体模型名称允许配置，但业务代码不能硬编码。
- 音频格式由配置决定，默认使用 `wav`。
- 豆包新版控制台接入必须使用 `CLASS_UP_DOUBAO_API_KEY` 环境变量名，不得保存 API Key 原文。
- 豆包标准版只接受公网音频 URL；本项目通过 SFTP 上传分段音频到云服务器后再调用豆包。
- `config.example.yaml` 必须可以直接作为无密钥 mock 流程的配置模板使用。
- Web UI 选择豆包时，页面只填写豆包 API Key；SFTP 主机、用户名和私钥路径仍从 `.env` 或系统环境变量读取。
- Web UI 选择豆包时，后端默认填充 `endpoint=https://openspeech.bytedance.com`、`model=bigmodel`、`resource_id=volc.seedasr.auc`，并将上传 provider 切换为 `sftp`。

## 5. Manifest 要求

M1 的数据结构、状态枚举、路径规则和阶段交接以 `docs/数据协议.md` 为准；本计划只描述 M1 实施任务，不另行定义冲突结构。

`manifest.json` 至少记录：

- 输入视频绝对路径
- 输出目录绝对路径
- 配置摘要，不包含密钥原文
- 阶段状态
- 音频文件路径
- 分段列表
- 每个分段的状态、音频路径、转录结果路径、错误信息
- 最终字幕和文稿路径

阶段状态取值：

- `pending`
- `running`
- `success`
- `failed`
- `skipped`

断点恢复要求：

- 已完成分段不重复调用转录服务。
- 失败分段允许单独重试。
- 合并阶段只能读取 `status == "success"` 且有 `transcription_path` 的分段结果。
- 如果分段因文件大小超限被重切，父分段必须标记为 `superseded`，新的子分段写入独立 `index` 和 `parent_segment_id`，避免断点恢复时同时读取旧分段和新分段。
- 所有可参与合并的分段必须保持 `index` 连续递增。

## 6. 执行步骤

### Step 1：初始化工程骨架

任务：

- [ ] 创建 `src/class_up/` 包结构
- [ ] 创建基础 `__init__.py`
- [ ] 创建 CLI 入口文件
- [ ] 创建配置、manifest、media、transcription、utils 模块

验收：

- Python 能正常导入 `class_up`
- CLI 能输出帮助信息
- CLI 提供 M1 音频准备命令和 M1 完整流水线命令，旧式 M1 调用暂时兼容

### Step 2：配置加载与路径初始化

任务：

- [ ] 读取 YAML 配置
- [ ] 读取 `.env` 中的密钥环境变量
- [ ] 校验必填配置
- [ ] 创建标准输出目录
- [ ] 初始化 `manifest.json`

验收：

- 缺少必填配置时给出明确错误
- 不在日志和 manifest 中输出密钥原文
- 输出目录可以重复初始化且不会覆盖原视频

### Step 3：ffmpeg 封装

任务：

- [ ] 检查 ffmpeg 是否可执行
- [ ] 封装命令执行函数
- [ ] 捕获 stdout、stderr、退出码
- [ ] 将 ffmpeg 错误写入 manifest

验收：

- ffmpeg 不存在时能明确报错
- ffmpeg 执行失败时保留错误信息

### Step 4：音频提取与分段

任务：

- [ ] 提供 M1 视频转音频函数，输入视频路径和输出路径，输出标准化音频
- [ ] 用 ffprobe 获取视频时长，写入 manifest
- [ ] 从视频提取音频
- [ ] 标准化采样率和声道
- [ ] 按配置时长切段，保留配置化重叠区
- [ ] 切段后检查每段文件大小；若超过上传限制，自动将该段时长减半后重切，最多重试 2 次，仍超限则写入 manifest 并报错
- [ ] 超限重切后使用稳定子分段 ID，例如 `segment-0003a`、`segment-0003b`，并在 manifest 中记录 `parent_segment_id`
- [ ] 将分段列表（含起止时间、文件路径、文件大小）写入 manifest

验收：

- `audio` 命令可将视频转换为默认 16kHz 单声道 `wav`
- `audio` 命令支持 `--output` 指定完整输出文件路径
- `audio` 命令只执行 M1 音频准备中的转换动作，不触发分段或转录
- 完整 M1 流水线必须将标准化音频路径写入 manifest
- 分段文件存在且可被 ffmpeg 识别
- 分段起止时间连续，重叠区符合配置
- 所有分段文件大小不超过配置的上传限制
- manifest 记录视频总时长
- 超限重切不会留下可被后续阶段误读的旧分段记录

### Step 5：转录适配接口与 mock 服务

任务：

- [ ] 定义转录服务接口
- [ ] 定义统一分段转录结果结构，使用顶层 `items[]` 表达转录片段
- [ ] 实现 mock 转录服务
- [ ] 保存每段转录结果 JSON

验收：

- mock 服务可跑通所有分段
- 每段结果包含 `schema_version`、`segment_id`、`source_audio`、`time_base`、`items[]`、`error`
- `time_base` 固定为 `segment_relative`
- `items[].start` / `items[].end` 均为分段内相对时间，不直接写成原视频绝对时间
- 不把旧版顶层 `start`、`end`、`text` 结构作为内部真源

### Step 6：首个真实转录服务适配器

任务：

- [x] 根据配置选择转录服务适配器
- [x] 读取鉴权信息
- [x] 实现 SFTP 分段音频上传，生成公网 URL
- [x] 实现豆包新版控制台 submit/query 调用
- [x] 将豆包 `result.utterances[]` 转换为内部 `items[]`
- [x] 保存豆包原始响应到 `intermediate/transcription/raw/`
- [ ] 实现超时、重试和并发控制
- [ ] 记录每段转录状态

验收：

- 真实服务失败不会导致任务无记录退出
- 重跑时跳过已成功分段
- 并发数由配置控制
- manifest 和原始响应文件不得包含 API Key 原文
- 豆包返回毫秒时间戳必须转换为分段内相对秒
- 缺失 `utterances` 但存在全文文本时必须生成复核记录
- Web UI 选择豆包时必须填写 API Key，mock 模式允许留空

### Step 7：合并与字幕生成

任务：

- [ ] 按 manifest 中可参与合并分段的 `index` 升序读取转录结果后再合并，不按 `segment_id` 排序
- [ ] 修正每段时间戳偏移（加上该段在原始音频中的起始时间）
- [ ] 重叠区处理策略：默认保留后段重叠区，前段仅丢弃与后段时间重叠的尾部片段；禁止同时裁掉前段尾部和后段头部导致重叠区内容丢失
- [ ] 对无法安全判断的重叠区，优先保留内容并在 manifest 的统一 `reviews[]` 对象中记录复核需求
- [ ] 生成 `full_subtitle.srt`
- [ ] 生成 `full_transcript.txt`

验收：

- SRT 编号连续
- SRT 时间码单调递增，无负值
- 文稿按转录顺序输出
- 重叠区处理不得造成整段重叠内容缺失
- 无法安全去重的重叠区会写入 `reviews[]`，不使用散落布尔字段
- 合并结果路径写入 manifest

## 9. 风险与处理

| 风险 | 处理 |
|---|---|
| 转录服务时间戳格式不一致 | 适配器层转换为统一内部结构 |
| 单段音频超过上传限制 | 分段后检查文件大小，必要时缩短分段时长，并用稳定子分段 ID 更新 manifest |
| 豆包标准版只接受公网 URL | 通过 SFTP 上传分段音频到 `https://boneorbit.com/class-up/audio/` 后提交识别 |
| ffmpeg 命令失败 | 集中在 `media/ffmpeg.py` 捕获并写入 manifest |
| 重叠区重复文本难以完全去除 | 先实现保守去重，宁可保留重复，也不能丢失重叠区内容 |
| API 限流 | 并发数、重试次数、超时全部配置化 |
| 断点恢复误跳过失败段 | manifest 明确区分 `success`、`failed`、`skipped` |

## 10. 执行顺序

推荐按以下顺序推进：

1. Step 1 工程骨架
2. Step 2 配置加载与路径初始化
3. Step 3 ffmpeg 封装
4. Step 4 音频提取与分段
5. Step 5 mock 转录服务
6. Step 7 合并与字幕生成（先用 mock 服务闭环，验证合并逻辑）
7. Step 6 首个真实转录服务适配器（替换 mock，接入本地或远程模型服务）
8. 测试补齐与手工验收

先用 mock 服务闭环，把工程问题和模型服务问题分开排查，闭环后再替换为真实转录服务适配器。
